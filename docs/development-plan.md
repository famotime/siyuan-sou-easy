# 思源笔记插件开发计划：更友好的查找与替换

## 1. 开发目标

基于 `plugin-sample-vite-vue` 模板，在当前仓库内先完成可运行的 V1：当前文档内的 IDE 风格逐条查找替换。

## 当前实际进度（2026-03-07 代码审计）

- 总体判断：当前仓库已经跑通 V1 的最小可用闭环，代码层面已实现当前文档内查找、命中导航、高亮、替换当前、跳过当前、全部替换、基础设置和自动化测试骨架。
- 当前里程碑判断：`M1`、`M2`、`M3`、`M4` 已达成，`M5` 在功能上已基本达成；测试与发布验证已有进展，但手工安装回归仍未补齐。
- 主要实现偏差：当前搜索与替换主要基于编辑器 DOM 实时采集和 `dom` 更新，而不是本计划原先设想的 SQL 检索 + `markdown` / `kramdown` 精确更新。
- 主要欠账：手工回归与思源内安装验证仍未补齐；主题组件目录尚未真正复用。

## 2. 技术路线

### 2.1 总体架构

拆成五层：

- 插件入口层：注册命令、顶栏入口、生命周期、事件监听；
- 编辑器上下文层：识别当前激活文档与当前 protyle；
- 搜索引擎层：从文档块构建可搜索结果集并计算命中；
- 替换执行层：将单个命中映射回块并调用官方 API 更新；
- UI 展示层：悬浮查找替换组件、高亮与反馈。

### 2.2 推荐目录规划

建议在现有 `src` 下新增：

- `src/plugin/`：插件生命周期、命令、事件绑定；
- `src/features/search-replace/`：核心业务；
- `src/features/search-replace/model.ts`：结果模型与类型；
- `src/features/search-replace/search-engine.ts`：匹配计算；
- `src/features/search-replace/replace-engine.ts`：替换执行；
- `src/features/search-replace/context.ts`：当前编辑器上下文；
- `src/components/search-replace/`：查找替换 UI；
- `src/stores/`：面板状态；
- `src/utils/`：通用 API 封装与字符串工具。

## 阶段 1：编辑器上下文识别

> 进度标注（2026-03-07）：**基本完成**。插件已监听 `switch-protyle`、`loaded-protyle-static`、`destroy-protyle`，并可在运行时解析当前 `protyle`、`rootId` 和文档标题；文档切换、关闭后的状态刷新也已具备。当前实现更依赖即时查询，未单独维护计划中提到的 `blockId` 状态。

### 目标

可靠获取“当前激活文档”和当前编辑器上下文。

### 任务

- 监听 `switch-protyle`、`loaded-protyle-static`、`destroy-protyle`；
- 维护当前活动文档的 `rootId`、`blockId`、DOM 容器引用；
- 设计上下文失效与重建机制；
- 在多标签 / 分栏切换时保证上下文同步。

### 验收

- 文档切换后，插件知道当前操作目标；
- 关闭文档或切换编辑器时，旧状态被正确清理。

## 阶段 2：文档索引与搜索引擎

> 进度标注（2026-03-07）：**基本完成，但实现路径与原计划不同**。已实现可搜索块收集、块顺序遍历、普通文本 / 大小写 / 全词 / 正则匹配、非法正则报错、标准化结果结构，以及输入防抖；当前实现基于编辑器 DOM 提取文本，而不是 `/api/query/sql` 拉取块数据，缓存也尚未补齐。

### 目标

完成当前文档的块索引、搜索规则解析和命中列表计算。

### 任务

- 使用 `/api/query/sql` 按 `root_id` 拉取当前文档块；
- 建立块顺序与可搜索块白名单；
- 实现普通文本、大小写、全词、正则四类匹配逻辑；
- 输出标准化结果结构：块、偏移、预览、是否可替换；
- 对非法正则进行错误处理；
- 对输入增加防抖与缓存。

### 建议实现细节

- 先以 `content` 做候选筛选；
- 需要精确替换时再取 `markdown` / `kramdown`；
- 全词匹配使用边界判断，不直接依赖简单空格分词；
- 正则模式下单独做异常捕获。

### 验收

- 输入任意查询词后可以得到稳定、可排序的命中列表；
- 模式切换后结果即时刷新；
- 大文档下不会明显阻塞 UI。

## 阶段 4：命中定位与高亮

> 进度标注（2026-03-07）：**基本完成**。当前命中已可映射到 DOM、自动滚动到可见区域，并同时支持普通命中与当前命中的高亮；切换命中时会清理旧高亮。文档变化后的恢复策略目前主要是刷新结果并保留索引，尚不是更稳健的“重新锚定原命中”。

## 5. 关键技术决策

### 决策 1：V1 仅支持当前文档

原因：最贴近 VS Code 编辑器内查找体验，且风险最低。

### 决策 2：SQL 仅做检索，写操作统一走 API

原因：符合思源推荐边界，更安全。

### 决策 3：V1 不支持跨块命中替换

原因：底层数据模型与用户心智不一致，强行支持风险高且收益有限。

### 决策 4：复杂块采用白名单支持

原因：先保证常用文本块稳定，再逐步扩展。

## 6. 风险排期建议

- 第一优先级风险：当前命中定位与替换映射；
- 第二优先级风险：编辑器切换状态同步；
- 第三优先级风险：长文档性能；
- 第四优先级风险：复杂块兼容。

建议先完成“当前文档 + 普通文本块 + 单条替换”的最小闭环，再扩展高亮、设置与增强能力。
